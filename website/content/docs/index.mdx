browserlane is browser automation for **humans and AI agents**. It is one Rust
binary, `bl`, that exposes two surfaces over the same engine:

- A **CLI** for humans and scripts — `bl go`, `bl click`, `bl screenshot`,
  `bl find`, and dozens more.
- An **MCP server** for AI agents — `bl mcp` speaks JSON-RPC 2.0 over stdio, so
  an agent can drive a real browser through a catalog of tools.

Both surfaces drive Chrome over the **WebDriver BiDi** protocol.

## Why WebDriver BiDi

Most agent-facing browser tools sit on the **Chrome DevTools Protocol (CDP)** —
Chrome's vendor-specific side-channel, the same plumbing Playwright and Puppeteer
use underneath. browserlane instead drives Chrome **exclusively over
[WebDriver BiDi](https://w3c.github.io/webdriver-bidi/)**, the bidirectional,
W3C-standard protocol the browser vendors are converging toward. Building on the
standard rather than a vendor side-channel is the core bet.

## What makes it different

<Cards>
  <Card title="One static binary, no runtime">
    `bl` ships as a single native executable for macOS, Linux, and Windows.
    There is no Node or JVM runtime to install — download it, put it on your
    `PATH`, and run.
  </Card>
  <Card title="Visible browser by default">
    Commands open a real, visible Chrome window so you can watch automation
    happen. Pass `--headless` when you want it hidden.
  </Card>
  <Card title="One-command Chrome install">
    `bl install` downloads its own Chrome for Testing into a local cache — no
    system Chrome required. Run it once before your first browser command.
  </Card>
  <Card title="Two surfaces, one engine">
    The human CLI and the agent MCP server share the same browser engine, so an
    automation you prototype by hand behaves the same when an agent runs it.
  </Card>
</Cards>
